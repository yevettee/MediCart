## Codex에게 붙여넣을 요청문

현재 repo는 `/home/hi/MediCart/medicart_ws` 이고, ROS2 MediCart 프로젝트다. 반드시 현재 시스템 아키텍처 문서와 실제 패키지 구성을 먼저 읽고 진행해줘.

참고해야 할 문서:

- `/home/hi/MediCart/docs/architecture/05_multi_robot_system_architecture.md`
- `/home/hi/MediCart/docs/architecture/06_system_architecture_explanation_script.md`

핵심 목표는 `/robot6` 약 배달 모드를 dashboard에서 end-to-end로 동작하게 만드는 것이다.

최종 사용 흐름:

1. Dashboard에서 `약 배달 모드` 버튼을 누른다.
2. robot6이 undock 한다.
3. robot6이 약품실 좌표로 Nav2 자율주행한다.
4. 약품실 도착 후 OCR로 약품을 체크한다.
5. 사람이 약을 포장하고 포장지 겉에 QR을 붙인다.
6. Dashboard에서 보내야 되는 침대/병실을 선택해서 보낸다.
7. robot6은 YOLO tracking으로 간호사 또는 추종 대상을 따라간다.
8. 이동 완료 후 간호사가 포장된 약을 꺼내 QR 코드를 찍는다.
9. QR/OCR 결과를 처방전 DB와 비교한다.
10. 맞으면 DB에 투약 로그를 저장하고 dashboard는 성공 상태를 보여준다.
11. 틀리면 DB/화면 로그에 `해당 약품이 아닙니다`를 남기고 dashboard 화면을 적색 상태로 바꾼다.
12. 모든 약 배달 과정이 끝나면 docking station 좌표로 이동한 뒤 dock 한다.

tracking은 다른 조원이 작업 중일 수 있으므로, tracking 내부 구현을 크게 건드리기보다 현재 repo의 tracking 인터페이스와 조원 구현을 확인해서 최소한으로 연결해줘.

## 현재 분석 결과

새 패키지는 가능하면 만들지 말고 기존 패키지에 구현하는 것이 맞다.

역할 분담:

- `dashboard`: 약 배달 모드 버튼, 침대 선택, OCR/QR 검증 버튼, 빨간 에러 화면, 로그 표시
- `mission_manager`: robot6 약 배달 상태기 전체 오케스트레이션
- `db_bridge`: RTDB에서 방 좌표/처방 조회, 약 검증, 투약 로그 저장
- `scanner`: OCR 또는 QR 스캔을 받아 검증 요청
- `ocr_detector`: 실제 OCR 서비스
- `nurse_tracker`: 간호사/대상 추종. 조원 작업과 인터페이스만 맞춘다.

현재 좌표는 `src/dashboard/dashboard/dashboard_node.py`의 `DEFAULT_TARGETS`에 있다.

현재 dashboard 기본 좌표:

```text
101호 1번:
  x: -12.0
  y: -5.0
  yaw: -0.00143

101호 2번:
  x: -12.0
  y: -6.0
  yaw: -0.00143

102호 호출:
  x: -13.0
  y: -8.0
  yaw: -0.00143

약품실:
  x: -9.0
  y: -9.0
  yaw: -0.00143

Docking Station:
  x: -8.0
  y: -6.0
  yaw: -0.00142
  dock_after: true
```

새 맵을 따면 이 좌표와 yaw를 반드시 갱신해야 한다. 가능하면 RTDB `/rooms`에도 같은 좌표를 넣는 방향이 좋다.

권장 상태 전이:

```text
IDLE
  -> UNDOCKING
  -> MOVING_TO_PHARMACY

  -> WAITING_PACKAGE
  -> WAITING_TARGET_BED
  -> FOLLOWING_TO_BED
  -> WAITING_ARRIVAL_CONFIRM
  -> WAITING_PACKAGE_QR_SCAN
  -> VERIFYING_MEDICINE
  -> DELIVERED or ERROR
  -> RETURNING_HOME
  -> DOCKING
  -> IDLE
```

1차 구현에서는 tracking 완료를 자동 판단하지 말고 dashboard에 `도착 완료` 버튼을 두는 것이 안전하다.

## 구현 우선순위

### 1단계: dashboard에 약 배달 UI와 API 뚫기

목표:

- 버튼은 보이지만 실제 backend service가 없으면 에러를 잘 보여주게 만든다.
- 약 배달 flow를 사람이 dashboard에서 따라갈 수 있게 만든다.

필요 UI:

- `약 배달 모드 시작`
- `약품 OCR 확인`
- 침대 선택 버튼: 101호 1번, 101호 2번, 102호 호출
- `추종 시작`
- `도착 완료`
- `포장 QR 스캔/검증`
- `귀환 및 Dock`
- 상태 badge: 대기, 약품실 이동, 약품 확인, 추종 중, 검증 성공, 검증 실패

### 2단계: mission_manager에 약 배달 상태기 추가

권장 구현:

- 새 파일 `medication_delivery.py`에 상태기 클래스를 만든다.
- `mission_manager_node.py`는 service server와 상태기 tick만 연결한다.
- Nav2 action, Dock/Undock action은 `patrol_mode_node.py`와 dashboard 코드의 패턴을 참고한다.

주의:

- 현재 `mission_manager_node.py`는 이미 mode arbiter를 돌리고 있으므로 기존 기능을 깨지 말 것.
- callback 안에서 오래 blocking하지 말고 async action/result callback 또는 timer tick으로 상태를 전이할 것.

### 3단계: DB 처방 조회/검증 보강

먼저 순수 로직부터 만들 것:

- `load_prescription(fb, patient_id)`
- `medicine_from_record(record)`
- `verify_medicine_text(expected, scanned_text)`
- `parse_package_qr(payload)`

그 다음 service server에 연결:

- `GetPrescription`
- `VerifyMedicine`

테스트:

- fake FB로 prescription 읽기 테스트
- OCR keyword match 테스트
- QR payload match/mismatch 테스트

### 4단계: scanner/OCR 연결

처음에는 완전한 OCR이 없어도 mock/fallback으로 flow를 닫을 수 있게 만든다.

권장:

- `ocr_node`가 latest frame이 없으면 `success=false`
- OCR engine이 준비 안 되어 있으면 명확히 `OCR engine unavailable`
- scanner는 `VerifyMedicine` request를 받아 OCR 결과를 DB verify에 넘긴다.

### 5단계: tracking 연결

1차 구현:

- mission_manager가 `round` mode start를 걸면 tracking 시작
- dashboard `도착 완료` 버튼으로 `round` mode stop
- 자동 도착 판정은 나중에 추가

현재 mode start 방법:

```json
{
  "action": "start",
  "mode": "round",
  "params": {}
}
```

내부적으로는 `ModeArbiter.apply('start', 'round', params)` 또는 `/robot6/mission_request` 경유를 사용할 수 있다.

### 6단계: 통합 테스트

최소 통합 테스트 순서:

1. `ros2 action info /robot6/navigate_to_pose`
2. `ros2 action info /robot6/undock`
3. `ros2 action info /robot6/dock`
4. `ros2 service list | grep robot6`
5. dashboard에서 약 배달 시작
6. 약품실 이동 확인
7. OCR/QR 검증 API가 실패하더라도 실패 메시지가 dashboard에 표시되는지 확인
8. tracking start/stop topic 또는 mode 상태 확인
9. docking station 이동 후 dock 확인

## 구현 시 꼭 지킬 점

- 새 패키지는 만들지 않는다. 꼭 필요하면 먼저 이유를 설명한다.
- dashboard 좌표와 RTDB `/rooms` 좌표가 다르면, 1차 구현은 dashboard 좌표를 우선 사용한다.
- 새 맵을 딴 뒤에는 좌표뿐 아니라 yaw도 반영한다.
- DB 직접 접근은 가능하면 `db_bridge`에 몰아준다.
- scanner와 mission_manager는 RTDB 구조를 직접 알지 않게 한다.
- QR payload 형식이 확정되지 않았으면 JSON과 `patient_id|medicine_id|sequence_order` 둘 다 parsing 가능하게 만든다.
- tracking 조원 코드가 있으면 그 인터페이스를 먼저 읽고 맞춘다.
- 기존 `dashboard`의 수동 Nav2/dock 기능은 debug/manual 기능으로 유지한다.
- 기존 `mission_pool`/mode arbiter 기능을 깨지 않는다.

## 현재 가장 큰 미구현/리스크

1. `mission_manager`에 약 배달 service server가 없다.
2. `scanner_node`와 `ocr_node`가 placeholder다.
3. `prescription_server`가 아직 `medicines`를 빈 배열로 반환한다.
4. RTDB에 구조화된 `/prescriptions` 데이터가 실제로 있는지 확인해야 한다.
5. 포장 QR payload 형식이 아직 확정되지 않았다.
6. tracking 완료를 자동으로 알 수 있는 인터페이스가 없다.
7. 아키텍처 문서는 `/robot6/target_pose` 추종을 말하지만 현재 코드는 `/robot6/mode/round/cmd_vel` 기반이다.
8. 새 맵 좌표/yaw가 아직 반영되지 않았다.

## 내일 Codex가 바로 할 작업 지시

먼저 아래를 실행해서 현재 상태를 확인해줘.

```bash
git status --short --branch
rg -n "start_medication|scan_patient|scan_medicine|VerifyMedicine|GetOcrResult|ListRooms|round|mission_request" src
```

그 다음 이 순서대로 진행해줘.

1. `dashboard_node.py`, `index.html`, `app.js`, `styles.css`를 읽고 약 배달 UI/API를 설계한다.
2. `mission_manager_node.py`, `patrol_mode_node.py`, `prescription_session.py`, `mode_arbiter.py`를 읽고 약 배달 상태기를 어디에 붙일지 결정한다.
3. `db_bridge`의 `prescription_server.py`, `rooms_server.py`, `firebase_client.py`를 읽고 처방 조회/검증 로직을 보강한다.
4. `scanner_node.py`, `ocr_node.py`, `qr_scanner.py`를 읽고 OCR/QR 검증 흐름을 만든다.
